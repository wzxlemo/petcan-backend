# Petcan 硬件与软件功能映射文档

> 本文档定义"硬件前"与"硬件后"的功能区分，确保用户购买硬件后获得增量价值，而非推翻原有体验。

---

## 核心原则

| 原则 | 说明 |
|------|------|
| **音色不变** | 用户在 App 中生成的宠物音色，购买硬件后**完全保留**。硬件只是换个设备播放，不改变音色本身 |
| **数据延续** | 硬件购买前通过对话获得的数据，购买后**自动同步**到硬件数据源 |
| **功能增量** | 硬件带来的是**新增实时数据**，而非替代原有功能 |

| 功能 | 变化 | 数据来源 |
|------|------|---------|
| **宠物形象展示** | 不变 ✅ | 已生成的 2D 形象 |
| **宠物音色** | 不变 ✅ | 已确定的音色，硬件同步播放 |
| **互动播放** | 增强 | 硬件扬声器播放，App 远程触发 |
| **智能对话** | 增强 | 硬件传感器数据辅助小能分析 |
| **状态卡片** | ⬆️ **变为实时数据** | 硬件传感器 |

| • 睡眠 | ⬆️ 实时睡眠监测 | **项圈加速度 + 心率** |

| **异常提醒** | ⬆️ 新增 | 运动量过低/过高、睡眠异常自动推送 |

## 音色唯一性保障

**关键规则：每只宠物的音色是唯一的，一旦确定不再改变**

```
App 生成音色 → 存入数据库 (voice_settings.generated_voice_url)
                    ↓
            用户购买硬件
                    ↓
            硬件下载音色文件
                    ↓
            硬件播放 = 与 App 完全一致的声音
```

**技术实现：**
- 音色文件存储在 Supabase Storage
- 硬件通过蓝牙/Wi-Fi 同步音色文件
- 硬件固件只负责播放，不参与音色生成

## 对话页"小能"首次交互设计

**首次进入对话页时，小能主动询问：**

```
你好！我是小能，你的宠物健康管家 🐾

最近 [宠物名字] 状态怎么样？

我来帮你记录一下：
• 睡眠、饮食、排便都正常吗？
• 疫苗上次是什么时候打的？
• 最近有没有异常状况？

有任何问题都可以随时问我！
```

**小能根据用户回答提取信息，更新状态卡片：**

| 用户回答示例 | 提取信息 | 更新卡片 |
|-------------|---------|---------|
| "昨晚睡了10小时" | sleep: 10h | 睡眠卡片 → "10小时" |
| "今天遛了1小时" | exercise: 1h | 运动量卡片 → "充足" |
| "今天没怎么吃" | diet: poor | 当前状态 → "食欲不佳" |
| "疫苗去年打的" | vaccine: overdue | 今日提醒 → "疫苗待接种" |

## 技术栈对应

| 功能 | 当前实现 | 硬件后增强 |
|------|---------|-----------|
| 数据存储 | Supabase PostgreSQL | 同上 |
| 音色存储 | Supabase Storage | 硬件本地缓存 |
| 实时数据 | 无 | 硬件 MQTT → 后端 |
| 提醒推送 | 无（手动查询）| 微信服务通知 |
